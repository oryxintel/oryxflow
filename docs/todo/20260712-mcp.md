# oryxflow MCP server (read-only pipeline introspection)

> Corrects the record first: `20260622-sys-catalog.md` §10 and `20260622-sys-catalog2.md` §7
> reference an `oryxflow-mcp` repo — **no such repo exists**; those sections describe
> intent, not code. This plan is the from-scratch decision and spec. Sequenced **third**,
> after `20260712-engine-code-invalidation.md` (events) and `20260622-sys-catalog2.md`
> (catalog) — it is a thin surface over what those two build and has no value before them.

## Context — is MCP needed at all?

Honest starting point: **for the primary user today — a Claude Code agent working inside
the project repo — MCP adds almost nothing.** That agent has Bash/Read/Grep and a Python
environment; v4 deliberately makes its surface `oryxflow.events.status()/runs()` plus a
greppable `.oryxflow/events.jsonl`, taught by the plugin skill. Wrapping those in MCP tools
for that agent is redundant indirection, and the field-validated workflow (see v4's Field
validation section) never needed a server. If that were the only consumer, the answer would
be "not needed" and this file wouldn't exist.

MCP earns its existence from four things the Python-API-plus-files surface cannot do:

1. **Clients without code execution.** claude.ai web/desktop chat, mobile, Cursor-style
   chat panes, enterprise setups where agents get no shell. "What ran last night? Did the
   Houston backtest fail? What does `tilt_zscore` mean?" are exactly the questions a
   non-coding analyst (or a chat-only agent) asks — and they need a tool call, not a
   terminal. This is the population the data-trust catalog was designed for.
2. **Read-only permissioning.** `python -c "..."` and Bash cannot be safely auto-approved;
   a read-only MCP tool can. "The agent may *query* pipeline state but not *execute*
   anything" is a real policy in cautious/team settings, and MCP is the only surface that
   can express it.
3. **No-environment access.** The key design insight (below) is that the server can answer
   everything from the **data plane** — `.oryxflow/` events + index, `data/.oryxflow-code-status.json`
   state, catalog descriptors — **without importing the project's Python code at all**. So
   a session outside the project cwd, without the project venv, or spanning several
   projects can still ask "what's the state of pipeline X". No bootstrap, no `sys.path`
   games, no accidental execution of `tasks.py` top-level code.
4. **Protocol continuity with the hosted tier** (product plan, Oryx planning folder). Team
   agent memory is served via MCP; shipping a local server now fixes the tool contract —
   when the hosted endpoint exists, an agent's interaction pattern is unchanged (same tool
   names/shapes, org scope added), and the local server remains the free/offline tier of
   the same surface.

Verdict: **needed, but small, read-only, and third in sequence.** Not a dependency of v4 or
catalog2; a consumer of them. For in-repo Claude Code the skill remains the primary surface
and should say so.

### Design decisions

1. **Read-only in v1 — no run/reset/CRUD tools.** Mutating pipeline state from a chat
   client that can't see the code is the footgun class (a remote `reset()` deletes outputs;
   a remote `run()` executes arbitrary project code). Execution stays with code-execution
   agents. Revisit only with explicit per-tool user confirmation semantics, later.
2. **Data-plane only: the server never imports project task code.** Every v1 tool reads
   files the v4/catalog2 machinery already wrote: the events head + offloads, the SQLite
   index, the state JSON, catalog descriptors (which already embed schema, samples,
   semantics, lineage, `run_source`). Consequences: no venv coupling (the server needs read
   access to the project directory, not the project's interpreter), zero code-execution
   risk, and graceful degradation — a project without `enable_catalog()` still answers all
   run/history questions, catalog tools just return "catalog not enabled".
3. **Ships inside the oryxflow repo, not a separate repo.** *(Corrects the catalog plans'
   assumption.)* One maintainer, thin code, versioned in lockstep with the event/index
   schemas it reads: `oryxflow/mcp.py` + an `oryxflow[mcp]` extra (the `mcp` Python SDK as
   lazy optional dependency, matching the cloud-storage extras pattern) + a console entry
   point `oryxflow-mcp`. A separate repo can be extracted later if the surface grows;
   premature now.
4. **stdio transport, project root as configuration.** Launched per-project by the client
   (`.mcp.json` in the repo, or user-level config with `--project <path>`; multiple
   `--project` flags allowed, tools take an optional `project` argument when more than one
   is configured). No network listener in OSS v1 — that's the hosted tier.
5. **The tool contract is the future hosted contract.** Names, arguments, and result shapes
   below are frozen intentionally; the hosted endpoint implements the same tools with an
   org/project scope argument. Additive evolution only (same rule as event rows).
6. **Results are agent-shaped**: compact JSON, sizes bounded (samples capped as stored by
   the catalog; task_log messages truncated), timestamps ISO-UTC, and every result that
   references an artifact includes its **path** so a code-execution agent can hand off to
   `outputLoad`/pandas.

### Tools (v1)

| tool | args | returns | reads |
|---|---|---|---|
| `pipeline_status` | project? | `events.status()`: pending code warnings, last run per family, recent failures | index/JSONL |
| `recent_runs` | task_family?, flow?, last=20 | `task_ran`/`task_failed` rows (params, code_version, reason, duration, git_sha) | index |
| `pending_warnings` | project? | unacknowledged `code_warning`s (task, changed files, exits text) | index |
| `task_logs` | task_family?, last=50 | captured `task_log` messages with task context | index |
| `list_outputs` | project? | catalog artifacts: family, persist_key, path, shape, ts, status | catalog index |
| `describe_output` | task_family or task_id | full descriptor: columns, dtypes, profile, sample, `field_semantics`, code/schema hash | catalog index |
| `lineage` | task_id? | dependency edges down to raw-source citations | catalog index |

(Deliberately absent from v1: `run`, `reset`, `accept_code` — decision 1; `preview` —
requires importing task code, violating decision 2; free-text search over events — `jq` and
the hosted tier cover it.)

## Implementation

1. **`oryxflow/mcp.py`** — new module, stdlib + lazy `mcp` SDK import. A `FastMCP`-style
   server exposing the seven tools as thin wrappers: each resolves the project root
   (argument > single configured root), points `settings.eventspath`/`settings.dirpath` at
   it read-only, and calls the existing `oryxflow.events.*` / `oryxflow.catalog.*` readers.
   No logic beyond argument validation, bounding, and "catalog not enabled" degradation.
2. **Entry point** — `setup.py`: `extras_require['mcp'] = ['mcp']`,
   `entry_points={'console_scripts': ['oryxflow-mcp=oryxflow.mcp:main']}`; `main()` parses
   `--project` (repeatable) and serves stdio.
3. **Client config** — docs provide the `.mcp.json` snippet for Claude Code/Desktop
   (`{"mcpServers": {"oryxflow": {"command": "oryxflow-mcp", "args": ["--project", "."]}}}`)
   and note the extra install.
4. **Plugin skill note** — one paragraph in the plugin SKILL.md: in-repo Claude Code should
   prefer the Python API/JSONL (richer, composable); the MCP tools are for chat-only or
   out-of-repo contexts — don't double-wrap.
5. **Docs** — `docs/source/mcp.rst`: the why (condensed from Context), install, config,
   tool reference, read-only guarantee, data-plane-only guarantee (never executes project
   code).
6. **Catalog-plan reference fix** — the superseded-note in `20260622-sys-catalog.md` and
   catalog2 §7 point here instead of the nonexistent repo (one-line edits).

## Verification

1. **Baseline holds** (73 passing) — the module is import-inert without the extra;
   `pip install -e .` without `[mcp]` changes nothing.
2. **`tests/test_mcp.py`** (skipped when the `mcp` SDK is absent):
   - Tool functions called directly against a tmp project with a seeded event stream +
     catalog index: `pipeline_status`/`recent_runs`/`pending_warnings`/`task_logs` return
     the seeded facts; `list_outputs`/`describe_output`/`lineage` return descriptors, and
     return the "catalog not enabled" degradation when the catalog tables are absent.
   - **No-import guarantee**: place a `tasks.py` in the tmp project whose import raises —
     every tool still answers (proves data-plane-only).
   - Read-only guarantee: no tool call mutates `.oryxflow/` or `data/` (hash the trees
     before/after).
   - Multi-project: two `--project` roots, `project` argument routes correctly.
3. **End-to-end smoke** (manual): `pip install -e .[mcp]`; add the `.mcp.json` snippet to a
   real project (e.g. the submarket pipeline); in a **fresh Claude session outside the
   project venv**, ask "what ran recently and did anything fail?" and "what does column X
   in TaskY's output mean?" — answered via `recent_runs`/`describe_output` with no code
   execution.
